Section 3: Sending Engine + Unsubscribe

How emails get sent on schedule, and how contacts opt out

The Cron Endpoint — POST /api/cron/process-sequences

A single API route that any scheduler can call (Vercel cron, Upstash, a cron job, even a manual button for testing). Protected by a secret token in the header.

1. Find due enrollments
Query sequence_enrollments where status = 'active' and next_send_at ≤ now()
↓
2. Send email via Resend
Fetch step content, replace [First Name] tokens, send with unsubscribe link appended
↓
3. Log to email_logs
Record contact_id, sequence_step_id, resend_id, status = 'sent'
↓
4. Advance or complete
If more steps: increment current_step, set next_send_at = now() + delay_days. If last step: set status = 'completed'
⚡ Run frequency
Call this endpoint every 15–60 minutes. Emails send within one check window of their scheduled time. No emails are skipped — the processor catches up if it missed a window.

Unsubscribe Flow

Every sequence email gets an unsubscribe link in the footer. One click, no login required.

Footer appended to every sequence email:
---
You received this email from [Business Name].
Unsubscribe · [Business Name] · [City, State]
GET /unsubscribe?token=… page:
✅
You've been unsubscribed
You won't receive any more emails from [Business Name].
What happens on unsubscribe: Contact status → unsubscribed · All active enrollments → cancelled · Future sends skipped · Token is a signed JWT (no DB lookup needed to verify)
